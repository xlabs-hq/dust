# CEO Plan: Production-Ready Server + AI-Native Expansion

## Vision

### 10x Check
Make Dust the default state layer for connected applications and AI agents. MCP is already built. The expansion: AI agent state primitives, TypeScript SDK (targeting AI coding tools), zero-config `dust init`, webhooks, and full production observability. No one in the real-time sync space is addressing AI agent state. Dust is positioned to own this emerging market.

### Platonic Ideal
The user clones a project, runs `dust init`, and their app has shared reactive state. They write data in development, staging sees it. They deploy, production catches up in under a second. An AI agent updates config, every deployment reacts. They open the dashboard and see what changed, who changed it, when. They click "rollback" and time-travel. They add a teammate and they instantly see the same data. They never think about sync again. The feeling is invisibility.

Note: This vision is single-region initially. The architecture is designed to support future multi-region (edge reads, regional caches) without rework, but that is deferred until real user demand exists.

## Phased Delivery

### Phase 4-pre: Foundation Repair (Week 1-2)
Priority: Fix invariant violations that Phase 4 features would inherit and amplify.
These are bugs in the existing implementation, not new features.

1. **Fix counter/set event contract** — server broadcasts deltas (increment) and members (add/remove), but SDK treats them as materialized state. Server must broadcast materialized values.
2. **Fix file reference_count** — no decrement on overwrite, delete, rollback, or failed write. Add reconciliation path. Required for correct GC and billing.
3. **Fix rollback for ancestor ops** — a leaf created by an ancestor `set` is invisible to exact-path rollback queries. Rollback must trace through ancestor ops.
4. **Add SDK write-rejection handling** — SDK has no path to process `limit_exceeded` or `rate_limited` replies. Optimistic writes would stay stuck with no rollback.
5. **Build org-scoped REST API** — `dust init`, export, clone, and token management are blocked by the fact that store/token creation only exists behind browser-session controllers. Need machine-friendly API endpoints.
6. **Build catch-up completion protocol** — Crystal CLI sleeps 200ms hoping catch-up finished. Need end-to-end ack so "all clients caught up" can be computed for compaction.
7. **Extract ValueCodec + format_event helpers** — DRY fixes from code review.
8. **Fix 7 error/security gaps** — socket param validation, Writer error rescue, catch-up limit, rollback retention check, token-out-of-URL, token expiry enforcement, file store scoping.
9. **Expand maps into leaf entries on write** — match v4 design spec, correct billing model.

## Feature Specifications

### Log Compaction / Snapshots
- **Trigger:** Compaction runs when op count per store exceeds a configurable threshold (default: 10,000 ops) AND all connected clients have caught up past the compaction point.
- **Prerequisite:** Track per-connection `last_acked_seq` in channel state. Update on client ack or heartbeat. This does not exist today and must be built as part of this feature.
- **Mechanism:** Oban cron job checks stores hourly. For eligible stores: materializes current `store_entries` into a `store_snapshots` row, deletes `store_ops` with `store_seq <= snapshot_seq`.
- **Schema:** `store_snapshots` table: `{store_id, snapshot_seq, snapshot_data (jsonb), created_at}` (matches existing `StoreSnapshot` schema).
- **Rollback interaction:** Rollback is limited to ops after the latest snapshot. Attempting rollback before snapshot_seq returns `{:error, :beyond_retention}`. Deeper retention = deeper rollback window (tied to billing tier).
- **Catch-up sync:** If a reconnecting client's `last_store_seq` is before the snapshot point, server sends the snapshot + ops after `snapshot_seq`. This replaces the current "replay all ops" behavior.
- **Migration:** Additive. New `store_snapshots` table, no changes to existing tables.

### Billing / Limit Enforcement
- **Metered dimensions:** Stores (count), entries per store (leaf key count), file storage (bytes). Writes are not metered (they're small wire messages, not a meaningful cost driver).
- **Tiers:** Free (1 store, 1K keys, 100MB files, 7-day retention), Pro ($5/mo, unlimited stores, 100K keys/store, 10GB files, 30-day retention), Team ($10/user/mo, shared stores, 1M keys/store, 100GB files, 1-year retention).
- **Enforcement:** Hard reject on write when limit exceeded. `set` on new path checks key count, rejects with `{:error, :limit_exceeded, %{current: N, limit: M}}`. `set` on existing path always allowed. `delete` always allowed. `merge` checks net-new paths (keys in the merge map that don't already exist as entries); if adding them would exceed the limit, the entire merge is rejected.
- **Integration:** Stripe for payment processing. Billing state cached in `organizations` table (`plan` column). Limit checks are in-memory (read from cached org plan), not API calls.
- **Migration:** Add `plan` and `stripe_customer_id` columns to `organizations` table.

### MessagePack Wire Format
- **Approach:** Additive, content-type negotiation. Not a breaking change.
- **Mechanism:** WebSocket connection includes `format` param in query string (`?format=msgpack` or `?format=json`). Default remains JSON. Server encodes responses in the requested format.
- **Blast radius:** `dust_protocol` codec (already has Msgpax dependency), `StoreSocket` (format param handling), `StoreChannel` (encode/decode per connection), Crystal CLI (add MessagePack shard), TypeScript SDK (use msgpack-lite).
- **Implementation order:** Protocol lib codec → server socket/channel → Crystal CLI → TypeScript SDK.
- **Migration:** None. Existing JSON clients continue to work.

### Capability Versioning Negotiation
- **Mechanism:** Client sends `capver` in WebSocket connect params (already exists). Server responds with `{capver_min, capver_max, your_capver}` in join reply (already exists in design, partially implemented).
- **Behavior:** If client capver < server capver_min → reject with `{:error, :upgrade_required}`. If client capver > server capver_max → reject with `{:error, :server_outdated}`.
- **Capver 1:** All current operations. Capver 2: adds ephemeral stores + agent presence. This way existing clients don't break when new features ship.
- **Effort:** S (the socket already passes capver, just need the gating logic).
- **Migration:** None.

### AI Agent State Primitives
- **Ephemeral stores:** A store with `ttl` set at creation (`Dust.create("agent/session-123", ttl: :timer.hours(4))`). Oban job garbage-collects expired stores. Schema: add `expires_at` column to `stores` table. All existing stores have `expires_at: nil` (permanent). Ephemeral stores function identically to regular stores while alive.
- **Agent presence:** Single-node only (Phase 4C scope). Uses Phoenix.Presence on the store channel topic. Requires adding `{Phoenix.Presence, ...}` to the supervision tree, configured with existing `Dust.PubSub`. Connected clients can see which other clients (by device_id) are subscribed. Presence is ephemeral, not persisted. Multi-node presence (CRDT tracker) is deferred.
- **Context store type:** Deferred. Ship ephemeral stores with explicit TTL first. Let naming conventions emerge from real agent usage. Document recommended patterns in docs (e.g., `agent/{agent-id}/context`) rather than server-enforced behavior.
- **Migration:** Add `expires_at` column to `stores` table (nullable, additive).

### TypeScript SDK
- **Transport:** WebSocket (Phoenix Channel v2 JSON protocol, same as Crystal CLI). MessagePack optional.
- **Feature surface:** CRUD (get, put, merge, delete, enum), type operations (increment, add, remove), subscribe (on/off), status, rollback. Full parity with Elixir SDK.
- **Local cache:** Optional. In-memory by default (for AI agents and Lambda). SQLite adapter via better-sqlite3 for Node.js persistent use. No browser IndexedDB in v1.
- **Package:** `@dust-sync/sdk` on npm. Zero native dependencies in default mode.
- **Does NOT include:** MCP client (MCP tools are server-side). REST API wrapper (WebSocket is the transport).

### `dust init` Zero-Config Setup
- **Flow:** (1) Detect language from project files (mix.exs → Elixir, package.json → TS, Gemfile → Ruby, etc.), (2) Prompt for store name or generate from project name, (3) Check auth (run `dust login` if needed), (4) Create store on server, (5) Generate token, (6) Add SDK dependency to project manifest, (7) Generate config file with connection details, (8) Print "You're ready" with example code.
- **Languages supported in v1:** Elixir (mix.exs + config.exs), TypeScript (package.json + .env).
- **Implemented in:** Crystal CLI (`dust init` command).

### Webhook Notifications
- **Event types:** `entry.changed` (covers set, delete, merge, increment, add, remove, put_file). One event type, the `op` field distinguishes.
- **Registration:** API endpoint + CLI command. `dust webhook create --store james/blog --pattern "posts.*" --url https://example.com/hook`. Patterns use `DustProtocol.Glob` syntax (same as `dust enum`). `*` matches a single path segment; `**` matches zero or more segments.
- **Schema:** `store_webhooks` table: `{id, store_id, url, pattern, secret, active, created_at}`.
- **Delivery:** Oban worker. Retry with exponential backoff: 1m, 5m, 30m, 2h, 12h (5 attempts). After 5 failures, webhook marked inactive.
- **Security:** HMAC-SHA256 signature in `X-Dust-Signature` header using per-webhook secret.
- **Payload:** JSON, same shape as WebSocket event. `{store, store_seq, op, path, value, device_id, timestamp}`.

### Telemetry + Health Endpoints
- **/healthz:** Returns 200 if the Phoenix endpoint is up. No DB check.
- **/readyz:** Returns 200 if DB connection pool has available connections AND PubSub is responsive. Returns 503 otherwise.
- **:telemetry events:** `[:dust, :write, :start|:stop|:exception]` (latency, store_id), `[:dust, :connection, :join|:leave]` (connection count), `[:dust, :store, :entries]` (gauge, store size).
- **Structured logging:** Logger metadata includes `store_id`, `device_id`, `request_id` on all WebSocket operations.
- **Migration:** None. Pure additive code.

### Store Export/Import/Clone
- **Format:** JSON Lines (one entry per line). `{path, value, type, seq}` per line. Header line with store metadata.
- **Export:** `dust export james/blog > backup.jsonl`. Streams from server (GET `/api/stores/:name/export`). No size limit.
- **Import:** `dust import james/blog < backup.jsonl`. Streams to server (POST `/api/stores/:name/import`). Writes entries via normal write path (gets new store_seqs).
- **Clone:** `dust clone james/blog james/blog-staging`. Server-side: export + import in one Oban job. Returns a job ID. `dust clone --status <job-id>` polls completion. `--wait` flag blocks until done.
- **Conflict resolution on import:** Overwrites existing keys (LWW, import is "later").

### Rate Limiting per Token
- **Algorithm:** Token bucket (via Hammer library, already common in Phoenix ecosystem).
- **Default limits:** 100 writes/minute, 1000 reads/minute per token. Configurable per-organization (stored in org settings).
- **WebSocket behavior:** Rate limit check happens in `StoreChannel.handle_in/3` before dispatching to `Sync.write/2`. Extract `RateLimiter.check(token_id, :write)` function. Reads (enum, get) check `:read` bucket. Rate-limited writes return `{:error, %{reason: "rate_limited", retry_after_ms: N}}`. Connection stays open.
- **REST/MCP behavior:** HTTP 429 with `Retry-After` header.
- **Migration:** None. In-memory rate state (ETS).

### `dust status --watch`
- **Behavior:** Terminal UI refreshing every 1 second. Shows: connection state (connected/disconnected/syncing), stores with entry counts, current store_seq vs server seq (sync lag), recent operations (last 5).
- **Implementation:** Crystal CLI, using ANSI escape codes for in-place refresh.
- **Exits on:** Ctrl+C.

### `dust diff`
- **Store diff:** `dust diff store1 store2` — compares materialized state of two stores. Output: added/removed/changed keys with values.
- **Time-travel diff:** `dust diff store --from-seq 40 --to-seq 80` — shows changes between two sequence points. Replays ops in range.
- **Output format:** Colorized terminal diff by default. `--json` flag for machine-readable output.
- **Compaction interaction:** If `from_seq` is before the latest snapshot, the server returns `{:error, :compacted, %{earliest_available: N}}`. The CLI suggests `--from-seq <earliest>` or uses the snapshot state as baseline.
- **Implemented in:** Crystal CLI + server API endpoint (`GET /api/stores/:name/diff?from_seq=N&to_seq=M`).

## Outside Voice Findings (incorporated)

### Claude subagent (15 issues)

1. **Compaction seq race** — ACCEPTED. Store max seq in snapshot row. Read from snapshot when ops table is empty.
2. **Rollback write interleaving** — ACCEPTED. Writer GenServer acquires exclusive lock during store-level rollback.
3. **Export/import auth** — ACCEPTED. Bearer token auth (same as FileController) for new REST API endpoints.
4. **Map billing** — ACCEPTED (Approach A). Expand maps into leaf entries on write, matching the v4 design spec.
5. **`last_acked_seq` is a protocol change** — Acknowledged. This affects all SDKs and must be planned as cross-cutting work, not a compaction sub-task.
6. **Webhook fanout** — Noted. Pattern caching and write batching/debouncing needed at scale. Not blocking for v1.
7. **Clone backpressure** — Noted. Add batching and broadcast suppression to clone implementation.
8. **Compaction/webhook coordination** — Noted. Webhook payloads must be materialized before compaction deletes source ops.
9. **Phase 4A timeline** — Acknowledged as aggressive.

## Landscape Context

**Competitors:** Firebase, Supabase, PowerSync, ElectricSQL, Replicache/Zero, RxDB
**Dust's moat:** Multi-language SDK + pluggable cache adapter. Unique in offering server-agnostic sync where the client's local cache lives in whatever database the app already uses (Postgres via Ecto, SQLite, in-memory). Competitors like RxDB offer pluggable client storage but not multi-language server SDKs. ElectricSQL and PowerSync require Postgres on the server.
**Eureka:** AI agent state is an emerging, unaddressed market. Dust + MCP positions as the persistent state layer for AI agents.
